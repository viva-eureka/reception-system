## 1. プロジェクト概要

### 背景

現在、受付管理に「ラクネコ」（月額課金のiPadネイティブアプリ）を使用している。
月額コスト削減と、招待・カレンダー登録・当日受付フローの煩雑さを解消するために自社開発へ切り替える。

### 目的

| # | 目的 |
|---|---|
| 1 | ラクネコの月額費用を削減する |
| 2 | 招待作成〜来訪当日までのフローを自社業務に最適化する |
| 3 | Google Workspace（Calendar・Chat）との連携を自動化する |
| 4 | ISO27001に準拠したセキュリティ要件を満たす |
| 5 | 将来的な機能拡張の自由度を確保する |

### スコープ（フェーズ1）

- 受付端末（iPadキオスク） — PWA（Webアプリ）
- 管理コンソール（PC/iPad ブラウザ）
- Google Chat 通知
- 入退館記録

## 2. 機能要件

### 2-1. 受付端末（キオスク画面）

| ID | 機能 | 詳細 | 優先度 |
|---|---|---|---|
| F-01 | 待ち受け画面 | ロゴ・時計・日付・あいさつ文を表示。タップで受付開始 | Must |
| F-02 | 営業時間外表示 | 設定時間外は「ただいま営業時間外です」を表示 | Must |
| F-03 | **時間外受付の許可設定** | 設定でONにすると時間外でも受付操作を許可する（画面には時間外である旨を表示したまま） | Must |
| F-04 | **招待済みウェルカム表示** | 予約15分前〜開始時刻に「お打ち合わせ予約 ○○様」ボタンを待ち受けに表示 | Should |
| F-05 | 受付メニュー | 「配送業者」「ご予約・招待の方」の選択 | Must |
| F-06 | QR受付 | カメラでQRコードを読み取り、招待情報と照合して受付完了 | Must |
| F-07 | 名刺撮影受付 | カメラで名刺を撮影、OCRで氏名・会社名を抽出し確認・修正して受付 | Should |
| F-08 | キーボード受付 | 氏名・会社名をテキスト入力して受付 | Must |
| F-09 | **音声入力受付** | 端末マイクで氏名を音声入力（Web Speech API、端末依存） | Nice |
| F-10 | 配送業者ワンボタン | ボタン1回で配送受付完了・通知送信 | Must |
| F-11 | 受付完了画面 | あいさつ文を表示後、一定秒数で待ち受けに自動復帰 | Must |
| F-12 | 背景テーマ | 5種類のテーマを設定画面から選択可能（端末画面には非表示） | Nice |

|---|---|---|---|

| F-20 | 退館記録（手動） | 管理画面から任意のタイミングで退館記録を追加 | Must |

## 3. 非機能要件

### 3-1. ユーザビリティ

| # | 要件 |
|---|---|
| U-01 | 来訪者が初見で迷わず操作できること（ガイドテキスト・大きなボタン） |
| U-02 | 受付完了まで3タップ以内を目標 |
| U-03 | iPadのソフトキーボード表示時にレイアウトが崩れないこと |
| U-04 | 管理コンソールはPC（1280px以上）とiPad（768px以上）の両方で使用できること |
| U-05 | 体温測定機能は不要（コロナ禍対応機能は搭載しない） |

### 3-2. パフォーマンス

| # | 要件 |
|---|---|
| P-01 | 受付完了まで体感3秒以内（ネットワーク遅延を除く） |
| P-02 | 初回ロード（iPad Safari）で5秒以内に操作可能になること |

### 3-3. 可用性・信頼性

| # | 要件 |
|---|---|
| V-01 | 受付時間内の稼働率99%以上を目指す |
| V-02 | Wi-Fi一時切断時でも待ち受け画面の表示は維持されること（Service Worker / キャッシュ） |
| V-03 | Google Chat通知失敗時はリトライまたはエラーログを残すこと |
| **V-04** | **Supabaseの自動バックアップを有効化し、復旧手順を文書化すること（ISO27001 可用性要件）** |

### 3-4. セキュリティ（ISO27001 対応）

| # | 要件 | ISO27001対応 |
|---|---|---|
| S-01 | 管理コンソールは認証必須（Supabase Auth）。受付端末は認証不要 | 認証強化 ✅ |
| S-02 | QRコードのトークンは予測不可能な一意の値（UUID等）を使用 | 個人情報保護 ✅ |
| S-03 | 受付完了画面への会社名表示はデフォルト無効 | 個人情報最小化 ✅ |
| S-04 | 来訪者の個人情報はSupabaseのRow Level Security（RLS）で保護 | 権限管理 ✅ |
| S-05 | 保存データの保持期間を設定し、期限超過データは自動削除または匿名化 | 保存期間 ✅ |
| **S-06** | **通信はHTTPS、Supabaseの保存データは暗号化（at rest）を利用** | データ暗号化 |
| **S-07** | **受付端末はアクセスガイド（Guided Access）で固定し、一定時間操作なしで待ち受けに自動復帰** | 端末保護 |
| **S-08** | **管理コンソールの操作（登録・変更・削除・閲覧）をSupabaseのログで追跡可能にする** | 操作ログ |
| **S-09** | **IPアドレス制限を管理コンソールに設けることを検討（社内IPのみアクセス可）** | アクセス制限 |

### 3-5. 監査対応

| # | 要件 |
|---|---|
| **A-01** | **入退館記録をCSVエクスポートできること（期間・担当者・種別でフィルタ可）** |
| **A-02** | **管理画面の操作ログ（誰がいつ何を変更したか）を保持・エクスポートできること** |
| **A-03** | **データ保持期間の設定と、期限到来後の自動削除機能を備えること** |

### 3-6. 保守性

| # | 要件 |
|---|---|
| M-01 | フロントエンドとバックエンドを分離し、片方の変更が他方に影響しないこと |
| M-02 | GitHubで管理し、mainマージで自動デプロイ（Vercel）されること |
| M-03 | 設定（受付時間・テーマ等）はUIから変更できること（コード変更不要） |

## 4. システム構成（案）

```
┌─────────────────────────────────────────────┐
│                  クライアント                     │
│  受付端末 (iPad)          管理コンソール (PC)    │
│  PWA / Safari（Guided Access）  ブラウザ        │
└──────────────┬──────────────────┬────────────┘
               ▼                  ▼
┌─────────────────────────────────────────────┐
│              Next.js (Vercel)                 │
│  受付画面 / 管理画面 / API Routes             │
└──────────────┬──────────────────┬────────────┘
               ▼                  ▼
┌─────────────┐          ┌──────────────────────┐
│  Supabase   │          │  外部サービス           │
│  - Postgres │          │  - Google Chat         │
│  - Auth     │          │  - Google Calendar     │
│  - Storage  │          │  - OCR (Vision API)    │
│  - Realtime │          │  - メール送信 (Resend) │
│  - 操作ログ │          └──────────────────────┘
└─────────────┘
```

### 採用技術（案）

| 役割 | 技術 | 理由 |
|---|---|---|
| フロントエンド | Next.js + React + Tailwind CSS | 管理画面と受付端末を同一コードベースで管理 |
| バックエンド / DB | Supabase | Auth・RLS・Realtime・自動バックアップが揃っている |
| ホスティング | Vercel | GitHub連携で自動デプロイ。無料枠あり |
| メール送信 | Resend | 月3000通まで無料。QR添付対応 |
| カレンダー連携 | Google Calendar API（サービスアカウント） | 会議室リソース予約に管理者権限が必要 |
| 名刺OCR | Google Cloud Vision API | 日本語名刺の認識精度が高い |
| 通知 | Google Chat Incoming Webhook | 既存利用中 |

## 6. 未決定事項・検討点

| # | 項目 | 内容 |
|---|---|---|
| TBD-01 | 名刺OCR精度 | Google Vision APIの実精度は実機検証が必要。精度不足時は手入力フォールバック |
| TBD-02 | iPadキオスク運用 | Guided Access（1台）かMDM（複数台管理）の選択。現状1台なのでアクセスガイドで十分 |
| **TBD-03** | **データ保持期間** | 入退館記録の保持期間を決定する必要あり（個人情報保護・ISO27001）。案: 1年後に自動削除または匿名化 |
| TBD-04 | カレンダー連携の権限 | サービスアカウントにドメイン全体の委任が必要。Google Workspace管理者との調整が必要 |
| **TBD-05** | **退館フロー** | 自動退館の時刻設定で運用するか、手動のみにするか。ラクネコは有効状態 |
| **TBD-06** | **ノーショー通知の要否** | 未来訪の招待に対して担当者へ自動アラートを送るか否か |
| **TBD-07** | **音声入力の実装タイミング** | Web Speech APIはiOS Safariで利用可。端末マイク権限・精度の実機検証が必要 |
| **TBD-08** | **操作ログの保持期間・閲覧権限** | ISO27001上は変更ログの保持が必要。何年分保持するか、誰が閲覧できるかを決定 |
